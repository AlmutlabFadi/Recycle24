# TECH_DEBT_REGISTER

## الهدف
هذا السجل يوثق الديون التقنية المعروفة بدل إخفائها أو تجاهلها.

---

## طريقة التسجيل

لكل بند يجب تسجيل:

- ID
- العنوان
- النوع
- الشدة
- الحالة
- الموقع
- الوصف
- الخطر
- الإجراء المطلوب
- أولوية المعالجة

---

## البنود الحالية

### DEBT-001
- العنوان: lint غير نظيف
- النوع: جودة كود
- الشدة: متوسطة
- الحالة: مفتوح
- الموقع:
  - `src/realtime/gateway.ts`
  - تحذيرات متفرقة أخرى
- الوصف:
  توجد أخطاء lint حالية لا تتعلق مباشرة بإصلاح RBAC الأخير لكنها تمنع اعتبار الجودة منتهية بالكامل.
- الخطر:
  انخفاض الانضباط وتراكم استثناءات مستقبلية.
- الإجراء المطلوب:
  تنظيف أخطاء eslint الفعلية وترك التحذيرات غير الحرجة فقط إن لزم.
- أولوية المعالجة: عالية

### DEBT-002
- العنوان: لا يوجد typecheck script
- النوع: بنية أدوات
- الشدة: متوسطة
- الحالة: مفتوح
- الموقع:
  - `package.json`
- الوصف:
  المشروع لا يحتوي script صريحا لفحص TypeScript.
- الخطر:
  احتمال مرور أخطاء نوعية لا تظهر لأن build الحالي يتجاوز validation of types.
- الإجراء المطلوب:
  إضافة script واضح لـ typecheck وتشغيله ضمن بوابات الجودة.
- أولوية المعالجة: عالية

### DEBT-003
- العنوان: مشاكل encoding عربية في بعض الملفات
- النوع: جودة بيانات / نصوص
- الشدة: متوسطة
- الحالة: مفتوح
- الموقع:
  - بعض ملفات `rbac`
  - بعض النصوص في schema أو routes
- الوصف:
  توجد نصوص عربية محفوظة بترميز خاطئ وتظهر كرموز مشوهة.
- الخطر:
  رسائل سيئة للمستخدم وصعوبة search/maintenance.
- الإجراء المطلوب:
  مراجعة الملفات المتأثرة وتوحيد UTF-8.
- أولوية المعالجة: متوسطة

### DEBT-004
- العنوان: لا يوجد instrumentation أداء منهجي
- النوع: أداء / رصد
- الشدة: متوسطة
- الحالة: مفتوح
- الموقع:
  - admin endpoints
  - server logging
- الوصف:
  تم تحليل بعض endpoints لكن لا يوجد بعد نظام قياس رسمي لزمن التنفيذ والاستعلامات.
- الخطر:
  صعوبة تتبع regressions مستقبلية.
- الإجراء المطلوب:
  إضافة perf/logging baseline للمسارات الحرجة.
- أولوية المعالجة: عالية

### DEBT-005
- العنوان: فصل الطبقات ما زال جزئيا
- النوع: معمارية
- الشدة: متوسطة
- الحالة: مفتوح
- الموقع:
  - عدة `route.ts`
- الوصف:
  ما زالت أجزاء من business logic موجودة داخل routes بدلا من services/repositories.
- الخطر:
  صعوبة الصيانة والاختبار مع توسع المشروع.
- الإجراء المطلوب:
  refactor تدريجي للمسارات الحساسة.
- أولوية المعالجة: عالية

## قواعد التعامل مع الديون التقنية

1. لا يضاف debt جديد دون تسجيله هنا.
2. لا يستخدم debt الحالي كذريعة لإضافة debt جديد.
3. البنود العالية الأولوية تراجع قبل أي توسيع feature كبير.
4. debt الأمني أو المالي لا يبقى مفتوحا بلا خطة معالجة.
